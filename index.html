GitHub Pages in 30 secondi

🌐
Hosting statico

Serve file HTML, CSS, JS e asset statici, non backend Express o API server-side.

🔒
HTTPS incluso

Le pagine GitHub Pages supportano HTTPS e anche domini personalizzati correttamente configurati.

⚙️
Build moderna

Con Vite il deploy passa normalmente da GitHub Actions perché prima bisogna generare la build.

TIPS: GitHub Pages è perfetto per portfolio, demo frontend, documentazione e piccoli progetti pubblici.

Quando usarlo e quando no

✅ Ideale per

  • Portfolio personali e landing page
  • Progetti HTML/CSS/JS statici
  • Build React/Vue/Vite già compilate
  • Documentazione tecnica e demo pubbliche
  • Condivisione rapida con recruiter o clienti

❌ Non adatto per

  • Backend Node/Express in esecuzione
  • Server-side rendering e funzioni server
  • Runtime con variabili d'ambiente lato server
  • Applicazioni che dipendono da processi long-running
  • Siti oltre i limiti di Pages o con build troppo pesanti

Demo 1 — sito statico semplice

Passi in aula

  1. Creo repo pubblico su GitHub.
  2. Carico index.html, CSS e asset.
  3. Apro Settings → Pages.
  4. Scelgo branch e cartella root.
  5. Aspetto il deploy e apro l'URL.
git init git add . git commit -m "first deploy" git branch -M main git remote add origin https://github.com/USERNAME/nome-repo.git git push -u origin main

Demo 2 — app React

Concetto chiave

Su GitHub Pages non mandi il codice sorgente “così com'è”: prima costruisci la versione pronta da pubblicare.

npm install npm run build # output atteso: dist/

Focus:

  • Differenza tra src/ e dist/
  • Perché la build crea file ottimizzati
  • Perché GitHub Pages pubblica file statici

Configurazione Vite necessaria

// vite.config.js import { defineConfig } from 'vite' export default defineConfig({ base: '/nome-repo/' })

Perché è importante

Se pubblichi su https://username.github.io/nome-repo/, Vite deve sapere che gli asset partono da quella base.

Se il base è sbagliato, CSS e JS spesso vanno in 404 e la pagina appare rotta o vuota.

Workflow consigliato con GitHub Actions

Passi logici

  1. Push su main
  2. GitHub Actions installa dipendenze
  3. Esegue npm run build
  4. Carica la cartella dist
  5. Pubblica la build su Pages
name: Deploy to Pages on: push: branches: [main] permissions: contents: read pages: write id-token: write
steps: uses: actions/checkout@v6 uses: actions/setup-node@v6 run: npm ci run: npm run build uses: actions/configure-pages@v6 uses: actions/upload-pages-artifact@v5 with: path: ./dist uses: actions/deploy-pages@v5

Tre errori tipici

404
Asset non trovati

Quasi sempre base path sbagliato o asset referenziati con percorsi assoluti.

🔁
Refresh su una rotta

Con router client-side il server statico non conosce le route applicative se non gestisci il fallback.

🧩
Funziona in locale, non online

In produzione cambiano path, build, cache e spesso anche il casing dei file.

Incidenza
Percorsi e base path sono il problema più frequente nelle prime pubblicazioni.

Live coding

Mini app proposta

Una piccola app React/Vite tipo "todo list".

  • Homepage con lista
  • Dettaglio semplice o filtro
  • Stile minimo ma leggibile
  • README con URL finale

Verifica

Pubblica il progetto e verifica:

  • la home si apre correttamente
  • gli asset statici sono caricati
  • l'URL è condivisibile
  • il link è inserito nel repository

Checklist finale

Il repository è pubblico e contiene il codice aggiornato.
La build locale funziona e genera la cartella dist.
Su Vite è stato configurato correttamente il base.
In Settings → Pages è stato scelto il metodo di deploy corretto.
L'URL finale si apre anche in navigazione anonima o su un altro dispositivo.
